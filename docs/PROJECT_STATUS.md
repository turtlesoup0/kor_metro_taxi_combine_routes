# 프로젝트 현황 문서

> 최종 업데이트: 2026-03-25

## 1. 프로젝트 개요

**목적**: 대중교통을 기반으로, 환승 과정에서 택시가 더 효율적인 구간을 찾아 혼합 경로를 제안하는 한국형 길찾기 엔진

**핵심 가치**: 기존 길찾기 서비스가 대중교통과 택시를 분리하는 것과 달리, 택시 개입이 실질적 시간/비용 개선을 가져오는 구간만 자동 식별하여 최적 혼합 경로 제안

---

## 2. 구현 완료 기능 (MVP)

### 2.1 기반 시스템
- [x] ODsay + Kakao API 통합 (대중교통 경로 + 택시 + 지오코딩)
- [x] 그래프 기반 경로 탐색 (노드=역, 엣지=교통수단)
- [x] Dijkstra 알고리즘 + 5가지 전략별 다양한 경로 발견
- [x] 복합 스코어링 (시간·비용·환승·도보·대기·피로도)
- [x] FastAPI 웹 서버 + Leaflet 지도 UI

### 2.2 지원 교통수단
| 모드 | 설명 | ODsay trafficType |
|------|------|------------------|
| subway | 지하철 | 1 |
| bus | 시내버스 | 2 |
| train | 열차 (KTX/SRT/무궁화/ITX) | 4 |
| express_bus | 고속버스 | 5 |
| intercity_bus | 시외버스 | 6 |
| taxi | 택시 (Kakao Navi) | — |
| walk | 도보 | 3 |

### 2.3 경로 탐색 전략 (graph.py)
| 전략 | 설명 | 예시 |
|------|------|------|
| S1 | 대중교통 전용 | 버스→지하철→버스 |
| S2 | 택시→대중교통 (first-mile) | 택시→지하철 |
| S3 | 대중교통→택시 (last-mile) | 지하철→택시 |
| S4 | 택시→대중교통→택시 | 택시→열차→택시 |
| S5 | 대중교통→택시→대중교통 (mid-route) | 열차→택시→시외버스 |

- [x] ODsay 미지원 지역 인근 거점 경유 fallback
- [x] 고배차 버스(30분+) 택시 대체 변형 제안

- [x] 경로 gap 자동 연결 leg 삽입
- [x] 택시 비용-편익 표시 (시간 절약, 분당 비용)
- [x] 사용자 옵션: 우선순위 프리셋, 도보/택시 거리 조절, 교통수단 허용

## 4. 알려진 이슈 및 검토 필요 사항

### 4.1 ODsay API 호출량 제한 (심각도: 높음)
**현상**: 1회 검색에 30~40회 API 호출 → ODsay 429 (Too Many Requests) 발생
**영향**: 택시 구간 대중교통 교차검증이 파이프라인 후반에서 실행되어 API 할당량 소진 상태에서 동작 → 교차검증 실패
**대안 검토**:
- ODsay 호출 최적화 (불필요 호출 제거, 캐싱)
- 카카오맵 대중교통 API 활용 (대안 API)
- 네이버 지도 API (길찾기 포함, 단 이용약관 확인 필요)
- ODsay 유료 플랜 (호출 제한 완화)
- 교차검증 대상을 first/last mile 구간으로 한정 (5km 이하 택시 구간만)

### 4.2 카카오맵 대중교통 API 검토 (심각도: 중간)
**배경**: ODsay 결과의 신뢰성에 대한 의문 제기
**검토 내용**:
- 카카오 모빌리티 API: 대중교통 경로 검색 지원 (https://developers.kakaomobility.com)
- 별도 API 키 필요 (현재 REST API 키와 다름)
- ODsay와 병행하여 결과 비교 후 주 API 전환 가능성 검토

### 4.3 택시 구간 대중교통 교차검증 미완성 (심각도: 높음)
**현상**: 택시를 추천할 때 같은 구간의 대중교통 대안을 확인하지 않음
**예시**: 화신로298→행신역 3km를 택시로 추천하지만, 실제로는 버스 850(27분, 1,650원)이 존재
**구현 상태**: 코드 프레임워크는 작성됨 (`_cross_validate_taxi_legs`), 단 ODsay 429 문제로 실제 동작 불가
**해결 방향**:
1. API 호출량 최적화 후 교차검증 활성화
2. 또는 ODsay 대신 카카오 대중교통 API로 교차검증
3. 기존 파이프라인에서 이미 수집한 ODsay 결과를 캐시하여 재활용

### 4.4 택시 체크 해제 시에도 택시 포함 경로 제시 (심각도: 중간)
**현상**: UI에서 "택시 구간" 체크 해제해도 택시가 포함된 경로가 여전히 나옴
**원인**: `allow_taxi=False`가 그래프 전략(S2~S5)은 차단하지만, `_supplement_first_last_mile()`에서 추가되는 택시 leg은 차단하지 않음
**해결**: `_supplement_first_last_mile()`에서 `allow_taxi` 파라미터 확인 필요

### 4.5 택시 직행 경로의 거리 제한 미반영 (심각도: 낮음, 부분 해결)
**현상**: 슬라이더 16km 설정인데 23km 전구간 택시 직행 추천
**원인**: 택시 직행 경로 생성 시 `max_taxi_m`을 미적용 → `max_taxi_m * 2` 허용 코드가 있었음
**상태**: `* 2` 제거 완료, 추가 검증 필요

## 5. 아키텍처 개선 검토 사항

### 5.1 API 호출 최적화
현재 1회 검색에 30~40회 API 호출은 과다. 개선 방안:
- **캐시 레이어**: 동일 출발/도착 좌표의 ODsay 결과를 메모리 캐시 (TTL 10분)
- **불필요 허브 검색 제거**: 반경 내 허브가 너무 많으면 상위 N개만
- **장거리 전용 호출 분리**: 30km 미만은 터미널 교차 검색 불필요
- **교차검증 지연 실행**: 최종 상위 3개 경로의 택시 구간에만 교차검증

### 5.2 대안 API 검토
| API | 장점 | 단점 |
|-----|------|------|
| ODsay (현재) | 풍부한 대중교통 데이터, 역/노선 세부정보 | 호출 제한 엄격, 무료 1000회/일 |
| 카카오 모빌리티 | 통합 플랫폼, 택시+대중교통 일원화 가능 | 별도 API 키 필요, 상세 중간역 정보 제한적 |
| 네이버 지도 | 한국 최대 지도 서비스, 정확한 대중교통 | 이용약관 제약 (상업적 사용 제한) |
| Google Directions | 글로벌 지원, 안정성 | 한국 대중교통 데이터 부정확, 고비용 |

### 5.3 스코어링 개선
- 현재 스코어링은 택시 비용을 대중교통과 동일 weight로 처리
- 택시 비용에 별도 가중치 추가 고려 (사용자가 택시비 민감도를 독립 조절)
- 피로도 지수에 "택시 탑승 = 피로도 감소" 요소 반영

    └── index.html     # 웹 UI (Leaflet 지도 + 경로 카드)
docs/
